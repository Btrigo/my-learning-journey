## 🧠 What I Learned / Key Takeaways
- **DNS delegation:** Hostinger stays as the registrar but Route 53 handles DNS resolution. When nameservers in Hostinger are set to the Route 53 NS values, any lookup for `brandontrigo.com` gets directed to AWS — Route 53 then answers with the correct records (ALIAS → CloudFront)
- **How the full request flow works:** Visitor → Route 53 (ALIAS) → CloudFront edge location → OAC signs the request → S3 returns the file → CloudFront caches and serves it. The visitor never touches S3 directly
- **CloudFront caching and invalidations:** CloudFront caches files at edge locations globally. Changes to S3 are not automatically reflected — you must create an invalidation (`/*`) to tell CloudFront to discard cached content and fetch fresh files from S3
- **OAC vs OAI:** OAI is the legacy method for giving CloudFront access to a private S3 bucket. OAC is the current AWS best practice — it uses Signature V4 signing and is more secure. The bucket policy must match whichever method is actually attached to the distribution
- **S3 file path = URL path:** S3 has no real directories — it uses key prefixes. CloudFront requests files by their exact key. If `index.html` is at `brandontrigo.website/index.html` in S3 but CloudFront requests `/index.html`, S3 returns nothing
- **REST endpoint vs website endpoint:** OAC requires the S3 REST endpoint (`bucket.s3.region.amazonaws.com`). Static website hosting is only needed with the website endpoint — it is not needed and should be disabled in this architecture
- **ALIAS records:** DNS CNAMEs cannot be used at the zone apex (root domain). Route 53 ALIAS records solve this — they behave like CNAMEs but are allowed at the root and point directly to AWS resources like CloudFront
