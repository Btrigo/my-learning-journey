## 🧠 What I Learned / Key Takeaways
- Security groups vs. NACLs: security groups are stateful and instance-level (allow-only), NACLs are stateless and subnet-level (support allow and deny, rules evaluated in order)
- IAM roles are temporary, assumable permission sets gated by a trust policy — same underlying pattern whether the "who" is GitHub Actions (Project 03) or an AWS service like EC2 (today's lab); an IAM instance profile is the wrapper needed to attach a role to an EC2 instance specifically
- Terraform's `plan` (and CloudFormation's Change Sets) exist to preview changes before committing — same safety principle across IaC tools
- Terraform has no automatic rollback on a failed `apply`, unlike CloudFormation, which will auto-revert a failed stack update — a real architectural tradeoff, not just a style difference
- Reaffirmed (with better reasoning than the material itself provided) why Terraform beats CDK/CloudFormation for job-market reach: cloud-agnostic, larger community/module ecosystem, broader posting share even in AWS-centric shops
