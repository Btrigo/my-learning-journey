## 🧠 What I Learned / Key Takeaways
- `terraform import` only reads from the cloud provider into local state — it never writes to the provider. The only step that can actually change live infrastructure is `apply`, and only when the plan shows real proposed changes
- ARNs share a fixed 6-field skeleton, but which fields are populated (region, account ID) is decided per-service, not universal — some services are global (empty region), some don't even need an account ID
- Trust policy condition operators matter: `StringEquals` requires an exact match, `StringLike` supports wildcards — using `StringLike` with no actual wildcard in the value is a subtle unnecessary attack-surface widening, even if functionally identical to `StringEquals` in the moment
- Authentication (proving who you are) and authorization (what you're allowed to do) are separate checks — having valid credentials doesn't by itself grant permission; a separate policy lookup determines what you can actually do
- `.gitignore` and Terraform state have nothing to do with preventing duplicate cloud resources — that protection comes entirely from proper `import`, not from what is or isn't committed to version control
- Terraform's own official guidance is to commit the provider lock file (`.terraform.lock.hcl`) for reproducibility, even though the `.terraform/` binary folder and state files should never be committed
