## 🧠 What I Learned / Key Takeaways
- **IAM best practices**: Root account is for break-glass scenarios only. Day-to-day work is done through an IAM user with AdministratorAccess attached at the group level, not directly to the user. MFA enabled on both accounts.
- **S3 static website hosting**: Two distinct S3 endpoints exist — the bucket endpoint and the website endpoint. CloudFront must point to the **website endpoint** to correctly resolve index documents and handle routing.
- **DNS record types**: A (IPv4), AAAA (IPv6), CNAME (hostname alias, cannot be used on naked/root domain), MX (mail routing), TXT (domain verification, SPF/DKIM/DMARC), NS (nameservers), SOA (auto-generated, never touched manually), SRV (service location), and ALIAS (Route 53-specific, resolves hostname to IP internally — allows root domain to point at CloudFront).
- **Registrar vs DNS**: Domain registrar (Hostinger) and DNS provider are separate concerns. Registrar holds ownership of the domain. DNS answers queries. You can keep your registrar and delegate DNS to Route 53 by updating nameservers — no domain transfer required.
- **CloudFront as CDN**: Serves cached content from the nearest edge location globally. Data transfer from S3 to CloudFront is free. Egress from CloudFront to end users falls under the free tier (1TB/month). Effective monthly cost for this stack is ~$0.50 (Route 53 hosted zone only).
- **ACM certificates**: Must be requested in **us-east-1** regardless of where other resources live — CloudFront only reads certs from us-east-1. DNS validation preferred over email validation — auto-renews without manual intervention.
- **S3 objects**: Flat key-value store, not a hierarchical file system. Folder structure is simulated via key naming conventions. Objects are immutable — updates overwrite, not edit in place.
- **WAF**: Not applicable for fully static sites with no user input, backend, or database. Adds cost with no security benefit in this context.
