## What I Learned / Key Takeaways

### S3 Storage Classes
- **S3 Standard-IA** — retrieval is milliseconds. "Infrequent" refers to access frequency, NOT speed.
- **S3 One Zone-IA** — stores in only 1 AZ, cheaper than Standard-IA. Use when data is re-creatable and low latency is still needed. This is the cheapest option that still provides fast (millisecond) retrieval.
- **S3 Glacier Flexible Retrieval** — minutes to hours. NOT suitable when low latency is required.
- **S3 Glacier Deep Archive** — up to 12 hours. Lowest cost but slowest.
- **Trigger:** "cheapest + low latency + infrequent access + re-creatable data" = **S3 One Zone-IA** (not Glacier)

| Storage Class | Retrieval Time | AZs | Boot Volume |
|---|---|---|---|
| S3 Standard | Milliseconds | ≥3 | N/A |
| S3 Standard-IA | Milliseconds | ≥3 | N/A |
| S3 One Zone-IA | Milliseconds | 1 | N/A |
| S3 Glacier Instant Retrieval | Milliseconds | ≥3 | N/A |
| S3 Glacier Flexible Retrieval | Minutes–Hours | ≥3 | N/A |
| S3 Glacier Deep Archive | Up to 12 hours | ≥3 | N/A |

### EBS Volume Types
- **General Purpose SSD (gp2/gp3)** — most workloads, default choice, only SSD type usable as boot volume
- **Provisioned IOPS SSD (io1/io2)** — high performance databases, consistent IOPS
- **Throughput Optimized HDD (st1)** — big data, sequential workloads
- **Cold HDD (sc1)** — infrequent access, lowest cost
- **CRITICAL:** HDD volumes (st1, sc1) **cannot** be used as boot volumes

### Route 53 vs CloudFront

**TRAP:** "Serve dynamic web content to users globally" = **CloudFront**, NOT Route 53.

- **Route 53** = DNS — directs traffic to the right server (the GPS)
- **CloudFront** = CDN — actually delivers the content to users globally (the highway)

They work together but do different jobs. Route 53 without CloudFront can route users, but CloudFront is what delivers the content fast globally. You can't reach servers without DNS, but DNS alone doesn't serve content.

---

### Global Accelerator vs CloudFront

**TRAP:** These are commonly confused.

- **Global Accelerator** = routes traffic through AWS's private global network backbone → **decreased latency reaching your application**. Static IPs, anycast routing. Does NOT accelerate the EC2 instance itself — it accelerates the network path to it.
- **CloudFront** = CDN, caches content at edge locations, accelerates content delivery

**Trigger:** "decreased latency accessing applications" = Global Accelerator. "Cache/deliver content" = CloudFront.

### Instance Store vs EBS
- **Instance Store** = temporary block-level storage, physically attached to the host, fastest disk performance, **data lost when instance stops**
- **EBS** = persistent block storage, survives instance stop/start
- **Trigger:** "temporary" + "block-level" + "fastest performance" = Instance Store

---

### RDS Read Replicas vs Multi-AZ
- **Read Replicas** = offload read traffic, increase read throughput. Trigger: "read-heavy workloads"
- **Multi-AZ** = high availability, automatic failover. Trigger: "survive failures," "disaster recovery"
- **Standby replicas in Multi-AZ do NOT serve reads**

---

### Aurora Self-Healing
- **Amazon Aurora** explicitly advertises self-healing storage — automatically detects and repairs disk failures
- "Self-healing" + "high throughput" = Aurora (not DynamoDB)
- DynamoDB is serverless/auto-scaling but "self-healing" is Aurora's specific language

---

### AWS KMS vs CloudHSM
- **KMS** = managed key management, AWS manages the infrastructure, integrates with most AWS services. Use for encryption keys across AWS services.
- **CloudHSM** = dedicated physical hardware security module. You manage it. Use for: cryptographic operations, hashing, digital signing, SSL/TLS offloading, BYOL licensing compliance.
- **Trigger:** "cryptographic operations" + "hardware" + "digital signing" = CloudHSM. "Manage encryption keys" = KMS.

### AWS Root Account Best Practices
Verified against official AWS documentation (docs.aws.amazon.com/IAM/latest/UserGuide/root-user-best-practices.html):

- **Do NOT disable or fully delete the root account** — you will need it for account-level tasks only root can perform (closing account, certain billing tasks, recovering locked IAM admins)
- **Correct approach:** Lock it down — enable MFA, don't create access keys, don't use it daily
- **Create an IAM admin user** for daily tasks
- **Never create root access keys**
- In response to suspected unauthorized usage → **enable MFA** (not "disable root")

### Dedicated Hosts vs Dedicated Instances (BYOL)
- **Dedicated Instances** = your instances run on hardware not shared with other customers, but you don't control or see the physical host
- **Dedicated Hosts** = you control the actual physical server, visibility into sockets/cores — **required for BYOL (Oracle, Windows Server) licensing compliance**
- **BYOL always requires Dedicated Hosts**, not just Dedicated Instances

Central network hub that connects multiple VPCs and on-premises networks through a single gateway. "Hub and spoke" model — instead of VPC peering between every pair of VPCs, everything connects through one Transit Gateway.
**Trigger:** "connect multiple VPCs," "hub and spoke," "simplify network topology" = Transit Gateway.

### AWS Global Accelerator
Routes user traffic through AWS's private global network backbone to reduce latency. Provides static IP addresses anycast from multiple AWS edge locations.
**Trigger:** "decreased latency accessing applications globally" = Global Accelerator.
**NOT the same as CloudFront** — Global Accelerator is network layer, CloudFront is content delivery/caching.

### Availability Zones — Why Physically Separated
AZs are separated by meaningful distance (within 100km/60 miles of each other) specifically for **fault isolation** — if one AZ goes down due to disaster, the others are unaffected but close enough for low-latency replication.

---

### EC2 Billing — Per-Second
Amazon Linux, Ubuntu, Windows EC2 instances bill **per-second** with a 60-second minimum. 1 hour 30 minutes = billed for exactly 1 hour 30 minutes.

---

### IAM Roles — When to Use vs IAM Users
Use **IAM Roles** when:
- Outside entities (cross-account, third parties, federated users) need temporary access
- AWS services need permissions to act on your behalf (EC2 → S3, Lambda → DynamoDB)

Use **IAM Users** when:
- Employees need persistent, long-term access
- An administrator needs a permanent account
