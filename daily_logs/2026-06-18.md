## 🧠 What I Learned / Key Takeaways

**The "skeleton" mental model (core insight of the day):**
Almost every cloud infrastructure job — regardless of company or industry — is built from the same recurring core pieces: **auth, API layer, compute/server functions, database, container/build pipeline, distribution/networking.** Learning this skeleton deeply (rather than memorizing 12 unrelated AWS services) is what lets a job posting's wall of buzzwords be read as "oh, that's just the same stack again."

**The three loops on top of the skeleton:**
1. **Deploy loop:** Console (understand it manually) → Terraform/IaC (describe it as code) → CI/CD (automate the deployment)
2. **Operate loop:** Monitoring/observability (CloudWatch alarms/dashboards) → Security hardening (least-privilege IAM, locked-down SGs, encryption) → Scaling/resilience (load balancing, multi-AZ, auto-scaling)
3. **Packaging shift:** Containers (Docker) → Orchestration (Kubernetes/EKS) — a different unit of deployment that eventually wraps around both loops

My existing 12-project roadmap already walks this arc in order — Phase 1 = deploy loop, Phase 2 = operate loop + scaling, Phase 3 = containers/orchestration. The roadmap wasn't missing anything; I just hadn't named the unifying frame underneath it until today.

**DevOps clarified:** Not a discrete topic or phase to "complete" — it's the umbrella discipline (IaC + CI/CD + automation + monitoring + Dev/Ops collaboration) that's already woven through every project in the roadmap. There's no single "Project X: Learn DevOps" — by the end of Phase 1 I'll have hands-on reps in most of its core practices without a project literally labeled that.

**Terraform vs. AWS CDK vs. AWS SDK:**
- Terraform = cloud-agnostic IaC, written in HCL, declarative ("describe the end state")
- AWS CDK = AWS-only IaC, written in a real programming language (Python/TypeScript), compiles down to CloudFormation
- AWS SDK (boto3) = not IaC at all — a library used *inside running application code* to call AWS services at runtime (e.g., a Lambda function uploading a file to S3 that Terraform already created)
- Analogy: Terraform/CDK build the house; the SDK is the wiring/plumbing the people living in the house actually use.

**Cloud/Platform Engineer vs. Solutions Architect — career sequencing:**
Same technical skeleton, different job function. Engineer = builds/automates/operates a given architecture. Solutions Architect = decides which architecture fits a client's ambiguous, constrained, real-world situation and is accountable for that call — a skill earned through years of engineering experience, not built in isolation on solo portfolio projects (no real stakeholder, budget, or legacy constraint to push back). Correct sequencing: target Cloud/Platform/Support Engineer roles in 6 months; treat Solutions Architect as a 3–5 year horizon, with the AWS SAA cert coming *after* real job experience, not before.

**Job market patterns observed across today's postings:**
- **Clearance-gated roles** split into two real buckets: (1) "must already hold clearance" (Vantor, Leidos, Accenture Federal, BAH Platform Engineer) — not reachable without years + existing clearance; (2) "willing to sponsor, entry-level friendly" (BAH Cloud Engineer specifically) — large defense contractors often sponsor Secret clearance for junior hires, and existing Sec+ already satisfies the DoD 8570 IAT Level II requirement many of these list.
- **Azure-only postings are a real trap to watch for** — several "Cloud Engineer" titles today (Rogers Corp, SS&C/Calastone) were 100% Azure-specific with multi-year requirements; AWS depth doesn't transfer credibly against a 4-year Azure-specific bar. Worth tracking the AWS:Azure ratio going forward — if Azure shows up disproportionately in roles I want, AZ-900 (free, one weekend) becomes worth adding later.
- **Bare "Cloud Engineer" title (no Junior/Associate/I qualifier) consistently meant mid-to-senior** across every posting seen today — better search terms are "Cloud Support Engineer," "Associate Cloud Engineer," "Cloud Engineer I," "Junior Cloud Engineer," "Platform Engineer" (with caution — also skews senior at some companies).
