## 🧠 What I Learned / Key Takeaways

### 🗂 Amazon S3
- Object storage service; objects are stored in buckets scoped to a region but with globally unique names
- Storage classes: S3 Standard (frequent access), S3 Standard-IA (infrequent, AZ resilient), S3 One Zone-IA (single AZ, cheaper), S3 Glacier Instant/Flexible/Deep Archive (archival tiers by retrieval speed), S3 Intelligent-Tiering (auto-moves between tiers based on access patterns)
- Versioning: once enabled on a bucket, cannot be fully disabled — only suspended
- Replication: Cross-Region Replication (CRR) and Same-Region Replication (SRR); requires versioning enabled on both source and destination
- Lifecycle policies: automatically transition objects between storage classes or expire them after a defined period
- Encryption: SSE-S3 (AWS-managed keys), SSE-KMS (customer-managed keys via KMS), SSE-C (customer-provided keys), client-side encryption
- Pre-signed URLs: grant time-limited access to private objects without changing bucket permissions
- S3 static website hosting: enables hosting a static site directly from a bucket; bucket must be public or use CloudFront
- S3 Transfer Acceleration: uses CloudFront edge locations to speed up uploads over long distances
- Block Public Access settings: account-level and bucket-level controls that override any permissive bucket policies — critical for security hardening
- Object Lock: WORM (write once, read many) model; supports Governance and Compliance retention modes
- MFA Delete: requires MFA to permanently delete versioned objects or change versioning state

### 🌐 Amazon VPC
- Logically isolated network section within AWS; region-scoped, spans all AZs in that region
- Subnets: AZ-scoped; public subnets have a route to an Internet Gateway (IGW), private subnets do not
- Internet Gateway (IGW): allows public subnets to communicate with the internet; horizontally scaled and HA
- NAT Gateway: allows instances in private subnets to initiate outbound internet traffic; deployed in a public subnet; you need one per AZ for HA
- Route Tables: each subnet is associated with a route table that determines where traffic is directed
- Security Groups: stateful, instance-level firewall; return traffic is automatically allowed
- NACLs (Network ACLs): stateless, subnet-level firewall; inbound and outbound rules evaluated independently; rules evaluated in order by rule number
- VPC Endpoints:
  - **Gateway Endpoints** (free): for S3 and DynamoDB; traffic stays on AWS private network, added to route table
  - **Interface Endpoints** (PrivateLink, paid): for most other services; creates an ENI with a private IP in your VPC
- VPC Peering: connects two VPCs (same or different accounts/regions); non-transitive — traffic cannot flow through a peered VPC to a third VPC
- Transit Gateway: hub-and-spoke model; connects thousands of VPCs and on-prem networks; supports transitive routing
- Flow Logs: capture IP traffic metadata going to/from ENIs, subnets, or VPCs; written to CloudWatch Logs or S3

### 🔐 IAM
- Global service; not region-scoped
- Users: individual identities with long-term credentials (password + access keys)
- Groups: collections of users; policies attached to groups apply to all members
- Roles: assumed by trusted entities (EC2, Lambda, another AWS account, SAML identity provider); uses temporary credentials via STS
- Policies: JSON documents defining Allow/Deny on Actions, Resources, and Conditions; types include AWS Managed, Customer Managed, and Inline
- Permission Boundaries: set maximum permissions an identity can have; does not grant permissions on its own
- IAM Identity Center (formerly SSO): centralized access management for multiple AWS accounts via AWS Organizations
- Best practices: root account should only be used to create the first IAM user and enable MFA; use roles for applications, not access keys embedded in code
- Access Advisor: shows when services were last accessed; helps enforce least privilege
- Credential Report: downloadable CSV showing all users and status of their credentials

### 💻 Amazon EC2
- Virtual machines (instances) running on shared or dedicated physical hosts in AWS AZs
- Instance types: General Purpose (M/T), Compute Optimized (C), Memory Optimized (R/X), Storage Optimized (I/D), Accelerated Computing (P/G/Inf)
- Instance purchasing options:
  - On-Demand: pay per second/hour, no commitment, highest unit price
  - Reserved Instances (RI): 1 or 3-year commitment, up to 72% discount; Standard vs Convertible (can change instance family)
  - Savings Plans: flexible commitment ($/hour), applies to EC2, Lambda, and Fargate; Compute Savings Plans most flexible
  - Spot Instances: spare EC2 capacity, up to 90% discount; can be interrupted with 2-minute warning
  - Dedicated Hosts: physical server fully dedicated to you; needed for BYOL (Bring Your Own License) workloads
  - Dedicated Instances: run on hardware dedicated to your account; may share hardware with other instances from same account
- AMIs (Amazon Machine Images): template for launching EC2 instances; region-specific; can be created from running instances, shared, or purchased from Marketplace
- Placement Groups:
  - Cluster: all instances in one AZ, low latency, high throughput; high failure risk if hardware fails
  - Spread: each instance on distinct underlying hardware; max 7 per AZ; best for critical individual instances
  - Partition: instances divided into logical partitions, each partition on separate rack; good for distributed databases (HDFS, Cassandra, Kafka)
- User Data: bootstrap scripts that run once at instance launch (as root)
- Instance Metadata Service (IMDS): available at `http://169.254.169.254/latest/meta-data/`; IMDSv2 requires session-oriented requests (more secure)

### 📈 Auto Scaling Groups (ASG)
- Automatically adds/removes EC2 instances based on demand
- Launch Templates (preferred over Launch Configs): define what to launch (AMI, instance type, key pair, security groups, user data)
- Scaling policies:
  - Target Tracking: maintain a metric (e.g., CPU at 50%); simplest to configure
  - Step Scaling: scale by different amounts based on breach thresholds
  - Scheduled Scaling: scale at known times (e.g., every Monday 9 AM)
  - Predictive Scaling: uses ML to forecast load and scales proactively
- Cooldown period: default 300s; prevents ASG from launching/terminating before previous activity takes effect
- Lifecycle hooks: pause instance launch or termination to perform custom actions (e.g., install software, drain connections)
- Health checks: EC2 health checks (default) or ELB health checks; unhealthy instances are terminated and replaced

### ⚖️ Elastic Load Balancing (ELB)
- Application Load Balancer (ALB): Layer 7; routes based on HTTP/HTTPS content (path, host, headers, query strings); supports WebSockets; best for microservices
- Network Load Balancer (NLB): Layer 4; routes based on TCP/UDP; extreme performance, millions of requests/sec, static IP/Elastic IP per AZ; best for latency-sensitive workloads
- Gateway Load Balancer (GWLB): Layer 3; for inline inspection of traffic (e.g., firewalls, IDS/IPS); uses GENEVE protocol
- Classic Load Balancer (CLB): legacy; not recommended for new deployments
- Target Groups: route requests to registered targets (EC2, IPs, Lambda); ALB can route to multiple target groups
- Sticky Sessions (Session Affinity): routes a client to the same target using a cookie; supported on ALB and CLB
- Cross-Zone Load Balancing: distributes traffic evenly across all registered instances in all AZs; enabled by default on ALB, optional on NLB/GWLB

### 🌍 Route 53
- AWS's authoritative DNS service and domain registrar; global, highly available
- Record types: A (IPv4), AAAA (IPv6), CNAME (alias to another hostname, cannot be used for zone apex), Alias (AWS-native, can be used for zone apex, no TTL charge)
- Routing policies:
  - Simple: single resource, no health checks
  - Weighted: split traffic by percentage across multiple resources (A/B testing, blue/green)
  - Latency-based: route to region with lowest latency from user's location
  - Failover: active-passive; routes to secondary when primary health check fails
  - Geolocation: route based on user's geographic location; must define a default record
  - Geoproximity: route based on geographic location with optional bias to shift traffic toward/away from a region (requires Traffic Flow)
  - Multi-Value Answer: returns multiple healthy records; not a substitute for a load balancer
- Health Checks: monitor endpoints, other health checks (calculated), or CloudWatch alarms; used to route around failures
- Route 53 Resolver: answers DNS queries for VPCs; Resolver endpoints (inbound/outbound) enable DNS resolution between on-prem and VPC

### 🚀 AWS Global Accelerator
- Routes user traffic through AWS's global private network backbone to the nearest AWS edge location, then to application endpoints; reduces latency by avoiding the public internet
- Provides two static AnyCast IP addresses that remain fixed regardless of endpoint changes
- Supports failover across regions in under 30 seconds
- Key differentiator from CloudFront: Global Accelerator is for non-HTTP use cases (TCP/UDP) and scenarios requiring static IPs or fast regional failover; CloudFront caches content at edge

### ☁️ Amazon CloudFront
- CDN (Content Delivery Network); caches content at 400+ global Points of Presence (edge locations and regional edge caches)
- Origins: S3 buckets, ALB, EC2, any HTTP server
- Origin Access Control (OAC): restricts S3 bucket access to only CloudFront (replaces legacy OAI)
- Cache behaviors: define how CloudFront handles requests based on URL path patterns
- TTL: Default TTL 86400s (1 day); can be controlled via Cache-Control headers from origin
- Geo-restriction: allow or block access by country
- Field-level encryption: encrypt specific POST data at edge using public key; data remains encrypted all the way to your application
- CloudFront Functions vs Lambda@Edge: Functions run at edge (lightweight, sub-ms latency, JS only); Lambda@Edge runs at regional edge caches (Node.js/Python, more compute, supports request/response manipulation)

### 💾 EBS (Elastic Block Store)
- Persistent block storage attached to a single EC2 instance in the same AZ
- Volume types:
  - gp2/gp3: General Purpose SSD; gp3 allows independent IOPS/throughput provisioning
  - io1/io2: Provisioned IOPS SSD; for high-performance databases; io2 Block Express up to 256,000 IOPS
  - st1: Throughput Optimized HDD; for big data, data warehouses, log processing; cannot be boot volume
  - sc1: Cold HDD; lowest cost; infrequently accessed data; cannot be boot volume
- Multi-Attach: io1/io2 volumes can be attached to up to 16 Nitro-based EC2 instances in same AZ simultaneously
- Snapshots: incremental backups stored in S3 (managed by AWS); can be used to create volumes in any AZ or copy to another region
- EBS Encryption: uses KMS; encrypted volumes, snapshots, and data in transit between instance and volume are all encrypted; snapshots of encrypted volumes are encrypted

### 📁 Amazon EFS (Elastic File System)
- Managed NFS (Network File System); can be mounted concurrently to thousands of EC2 instances across multiple AZs
- Scales automatically; no capacity planning needed; pay for what you use
- Performance modes: General Purpose (default) vs Max I/O (higher throughput, slightly higher latency)
- Throughput modes: Bursting (scales with storage size) vs Provisioned (fixed throughput regardless of storage)
- Storage classes: EFS Standard and EFS Standard-IA; lifecycle management moves files between classes
- EFS only supports Linux; does not support Windows instances

### 🪟 Amazon FSx
- **FSx for Windows File Server**: fully managed Windows-native file system; SMB protocol; integrates with Active Directory; supports DFS namespaces
- **FSx for Lustre**: high-performance file system for HPC, ML, and video processing; can be linked to S3 for seamless data import/export; delivers up to hundreds of GB/s and millions of IOPS

### 🔄 AWS Snow Family
- Physical devices for large-scale data migration or edge computing when network bandwidth is insufficient
- **Snowcone**: smallest; 8TB HDD or 14TB SSD; up to 2 vCPUs; can use DataSync for online data transfer
- **Snowball Edge Storage Optimized**: 80TB usable storage; 40 vCPUs; for large data transfers
- **Snowball Edge Compute Optimized**: 42TB storage; 52 vCPUs + optional GPU; for edge ML inference and compute
- **Snowmobile**: exabyte-scale (up to 100PB per truck); for massive datacenter migrations
- All devices encrypt data at rest with 256-bit encryption using KMS; chain of custody tracked by AWS

### 🔁 AWS Storage Gateway
- Hybrid storage service bridging on-premises environments and AWS cloud storage
- **S3 File Gateway**: NFS/SMB interface; files stored as objects in S3; local cache for recently accessed data
- **FSx File Gateway**: provides local cache for FSx for Windows; for on-prem Windows apps
- **Volume Gateway**: presents iSCSI block storage backed by S3; **Stored mode** (primary data on-prem, async backup to S3); **Cached mode** (primary data in S3, cache on-prem)
- **Tape Gateway**: virtual tape library (VTL) interface for backup software; tapes stored in S3 and Glacier

### 🚚 AWS Migration Services
- **Application Migration Service (MGN)**: lift-and-shift; continuous replication of on-prem servers to AWS; minimal downtime cutover
- **Database Migration Service (DMS)**: migrates databases to AWS; source DB remains live during migration; supports homogeneous and heterogeneous migrations (use Schema Conversion Tool for heterogeneous)
- **Migration Hub**: central dashboard to track migration status across multiple migration tools
- **DataSync**: online data transfer service; agent-based; syncs data between on-prem NFS/SMB/HDFS and AWS storage (S3, EFS, FSx); supports scheduling and data integrity verification
- **Transfer Family**: fully managed SFTP, FTPS, FTP service; front-ends S3 or EFS; no infrastructure to manage

### 🔔 Amazon SNS (Simple Notification Service)
- Pub/sub messaging; topics fan out messages to multiple subscribers simultaneously
- Subscribers: SQS queues, Lambda functions, HTTP/HTTPS endpoints, email, SMS, mobile push
- Message Filtering: subscription filter policies allow subscribers to receive only relevant messages
- FIFO Topics: support ordered delivery and exactly-once processing; can only fan out to SQS FIFO queues
- SNS + SQS Fan-Out pattern: publish once to SNS topic, fan out to multiple SQS queues for parallel asynchronous processing

### 📬 Amazon SQS (Simple Queue Service)
- Fully managed message queuing; producers send messages, consumers poll and process, then delete them
- Standard Queue: at-least-once delivery, best-effort ordering, near-unlimited throughput
- FIFO Queue: exactly-once processing, first-in-first-out ordering, up to 3,000 msg/s with batching
- Visibility Timeout: period during which a received message is hidden from other consumers; default 30s, max 12 hours
- Dead Letter Queue (DLQ): messages that fail processing N times are sent here for isolation and debugging
- Long Polling: consumer waits up to 20s for messages; reduces empty responses and API calls vs short polling
- Message retention: 4 days default, max 14 days
- Max message size: 256KB (use S3 + SQS Extended Client Library for larger payloads)

### 📨 Amazon MQ
- Managed broker service for Apache ActiveMQ and RabbitMQ
- Use when migrating existing on-prem applications already using open messaging protocols (AMQP, MQTT, STOMP, OpenWire); not a replacement for SQS/SNS for new cloud-native apps

### ⚡ AWS Lambda
- Serverless compute; runs code in response to events without provisioning or managing servers
- Supported runtimes: Node.js, Python, Java, Go, Ruby, .NET, custom runtimes via Lambda Layers
- Execution limits: max 15 minutes per invocation, 10GB memory, 10GB ephemeral /tmp storage
- Concurrency: up to 1,000 simultaneous executions per region by default (soft limit); reserved concurrency guarantees capacity; provisioned concurrency keeps functions initialized to eliminate cold starts
- Triggers: API Gateway, ALB, S3 events, DynamoDB Streams, Kinesis, SNS, SQS, EventBridge, Cognito, etc.
- Deployment: zip packages (up to 50MB compressed / 250MB unzipped) or container images up to 10GB from ECR
- Lambda Layers: share common code/libraries across functions; up to 5 layers per function
- Lambda@Edge: run Lambda functions at CloudFront edge locations for request/response manipulation

### 📊 Amazon CloudWatch
- Monitoring and observability service
- Metrics: AWS services publish metrics automatically (CPU, NetworkIn, etc.); custom metrics can be published via API
- Alarms: trigger actions (SNS, ASG, EC2 actions) based on metric thresholds; states: OK, ALARM, INSUFFICIENT_DATA
- Logs: CloudWatch Logs Insights for querying log data; Log Groups → Log Streams hierarchy
- Dashboards: customizable visualizations of metrics
- EventBridge (formerly CloudWatch Events): serverless event bus; route events from AWS services, SaaS apps, or custom sources to targets (Lambda, SQS, SNS, Step Functions, etc.); supports scheduling via cron/rate expressions

### 🏗 AWS Step Functions
- Serverless orchestration of Lambda functions and other AWS services using visual state machines
- Workflow types: Standard (exactly-once, max 1 year, auditable) vs Express (at-least-once, max 5 minutes, high throughput)
- Use for multi-step workflows, human approval steps, error handling with retry/catch logic, and replacing complex Lambda chaining

### 🌱 AWS Elastic Beanstalk
- PaaS; deploy and manage applications without managing underlying infrastructure
- Handles provisioning, load balancing, auto-scaling, monitoring, and deployment
- Supports: Java, .NET, PHP, Node.js, Python, Ruby, Go, Docker
- Deployment policies: All at Once (fastest, downtime), Rolling, Rolling with Additional Batch, Immutable (new instances, safest), Blue/Green (swap environment URLs)
- You retain full control of the underlying AWS resources; Beanstalk is free (pay only for resources it provisions)

### 📡 Amazon Kinesis
- **Kinesis Data Streams (KDS)**: real-time data streaming; data retained 1-7 days (up to 365 days extended); consumers process data with KCL or Lambda; shards determine throughput (1MB/s in, 2MB/s out per shard)
- **Kinesis Data Firehose**: near real-time delivery to S3, Redshift, OpenSearch, Splunk, HTTP endpoints; fully managed, no consumer management
- **Kinesis Data Analytics**: run SQL or Apache Flink on streaming data in real time

### ⚡ ElastiCache
- Managed in-memory caching; reduces read load on databases
- **Redis**: supports persistence, replication (read replicas), Multi-AZ failover, pub/sub, sorted sets, geospatial indexes; can be used for session storage
- **Memcached**: simpler; multi-threaded; no persistence or replication; pure caching use case
- Common patterns: lazy loading (cache on miss), write-through (write to cache on every DB write), session store

### 🧊 Amazon MemoryDB for Redis
- Redis-compatible in-memory database with full durability (Multi-AZ transaction log); data survives restarts
- Not just a cache — a primary database; ultra-fast reads/writes with microsecond latency

### 📋 AWS CloudTrail
- Records API calls made in your AWS account (who, what, when, where); enabled by default, stores 90 days in console
- Trails: send events to S3 and/or CloudWatch Logs indefinitely; can cover all regions or a single region
- Event types: Management Events (control plane actions, e.g., creating an EC2 instance), Data Events (object-level S3 or Lambda invocations, off by default — extra cost), Insights Events (detect unusual activity)
- CloudTrail Lake: managed query service for CloudTrail events; SQL-based querying without needing Athena

### 🗄 Amazon RDS
- Managed relational database service; AWS handles OS patching, backups, hardware provisioning
- Supported engines: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Db2
- Multi-AZ Deployments: synchronous replication to standby instance in another AZ; automatic failover; standby cannot serve read traffic — it's purely for HA
- Read Replicas: asynchronous replication; can be in same AZ, cross-AZ, or cross-region; used for read scaling, not HA; can be promoted to standalone DB
- Automated Backups: daily snapshots + transaction logs; point-in-time restore up to 35 days
- RDS Proxy: connection pooling to reduce connection overhead; improves Lambda→RDS performance; handles failover transparently
- Encryption: at-rest with KMS (must be enabled at creation); in-transit with SSL/TLS; encrypted replicas require encrypted source

### 🌟 Amazon Aurora
- AWS-built cloud-native relational database engine compatible with MySQL and PostgreSQL
- Storage: distributed across 6 copies in 3 AZs automatically; self-healing; up to 128TB auto-scaling storage
- Up to 15 read replicas with sub-10ms replica lag; automatic failover to read replica in ~30s
- Aurora Serverless v2: scales capacity in fine-grained increments per-request; ideal for unpredictable workloads
- Aurora Global Database: one primary region + up to 5 secondary read-only regions; <1 second replication lag; supports managed planned/unplanned failover for DR
- Aurora Multi-Master: all nodes are read/write; for continuous write availability
- Backtrack: rewind database to a point in time without restoring from backup (MySQL-compatible only)

### 🍃 Amazon DynamoDB
- Fully managed serverless NoSQL key-value and document database; single-digit millisecond latency at any scale
- Capacity modes: Provisioned (specify RCU/WCU, can auto-scale) vs On-Demand (pay per request, automatically handles any traffic)
- DynamoDB Accelerator (DAX): in-memory cache for DynamoDB; microsecond read latency; write-through; no application code changes needed
- Global Tables: multi-region, multi-active; all regions are writable; last-writer-wins for conflict resolution
- DynamoDB Streams: ordered log of item-level changes; used with Lambda for event-driven architectures; retention 24 hours
- TTL (Time to Live): auto-delete items after expiry timestamp; no extra cost; good for session data, logs
- Point-in-time Recovery (PITR): continuous backups up to 35 days

### 🔵 Amazon DocumentDB
- MongoDB-compatible managed document database; designed to scale JSON workloads
- Similar architecture to Aurora: storage distributed across 3 AZs; 15 read replicas
- Not a true MongoDB fork — implements MongoDB APIs but not the same engine

### 🕸 Amazon Neptune
- Managed graph database; supports Property Graph (Gremlin) and RDF (SPARQL)
- Use cases: social networks, fraud detection, recommendation engines, knowledge graphs

### 📦 Amazon ECR (Elastic Container Registry)
- Fully managed Docker container image registry; public and private repositories
- Integrates natively with ECS, EKS, and Lambda container images
- Image scanning: basic (on-push) or enhanced (continuous, using Inspector)

### 🐳 Amazon ECS (Elastic Container Service)
- AWS-native container orchestration service
- Launch types:
  - **EC2 launch type**: you manage the EC2 instances (the container hosts); more control, lower cost at scale
  - **Fargate launch type**: serverless; AWS manages underlying infrastructure; you define CPU/memory; pay per vCPU and memory used
- Task Definitions: JSON blueprint defining container image, CPU/memory, environment variables, port mappings, IAM task role
- Services: maintain a desired count of running tasks; integrate with ALB for load balancing
- ECS Task Role vs ECS Task Execution Role: Task Role = permissions the container's application has; Task Execution Role = permissions ECS agent needs to pull images and push logs

### ☸️ Amazon EKS (Elastic Kubernetes Service)
- Managed Kubernetes control plane; AWS manages etcd, API server, and control plane HA
- Worker nodes: self-managed (EC2), managed node groups (EC2 with automated updates), or Fargate (serverless pods)
- Use when you need Kubernetes-native tooling, have existing K8s expertise, or need portability across clouds

### 🔑 AWS KMS (Key Management Service)
- Managed service to create and control encryption keys
- CMKs (Customer Managed Keys): you create, manage, enable/disable, and define rotation policy; $1/month per key
- AWS Managed Keys: AWS creates and manages for AWS services; automatically rotated every year; free to use
- AWS Owned Keys: AWS internal use; not visible to customers
- Key rotation: automatic annual rotation for CMKs (optional); keeps old key material for decryption; symmetric keys only
- KMS integrates with almost every AWS service for envelope encryption
- CloudHSM: dedicated hardware security module; you manage keys entirely; FIPS 140-2 Level 3 certified; no AWS access to keys

### 🛡 Security Services
- **AWS WAF**: web application firewall; protects against Layer 7 attacks (SQL injection, XSS, rate limiting); attached to ALB, API Gateway, CloudFront, AppSync
- **AWS Shield Standard**: automatically protects all AWS customers from Layer 3/4 DDoS at no cost
- **AWS Shield Advanced**: paid; enhanced DDoS protection; real-time attack visibility; cost protection; 24/7 DDoS Response Team (DRT)
- **AWS GuardDuty**: ML-based threat detection; analyzes CloudTrail, VPC Flow Logs, DNS logs, EKS audit logs; no agents needed
- **Amazon Inspector**: automated vulnerability scanning for EC2 (SSM agent required), Lambda functions, and ECR container images
- **Amazon Macie**: uses ML to discover and protect sensitive data (PII, PHI, credentials) in S3
- **AWS Security Hub**: aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer, and third-party tools into a single dashboard; automated compliance checks against CIS, PCI-DSS, etc.
- **Amazon Detective**: investigates security incidents; builds a graph model from CloudTrail, VPC Flow Logs, GuardDuty findings
- **AWS Artifact**: on-demand access to AWS compliance reports (SOC 1/2/3, PCI DSS, ISO); and AWS agreements (BAA, NDA)
- **ACM (AWS Certificate Manager)**: provisions, manages, and auto-renews TLS/SSL certificates; free for public certificates; integrates with ALB, CloudFront, API Gateway
- **AWS Secrets Manager**: stores and auto-rotates secrets (database credentials, API keys); native integration with RDS, Redshift, DocumentDB; charges per secret per month
- **AWS Directory Service**: managed Microsoft Active Directory in AWS; **Simple AD** (standalone LDAP), **AD Connector** (proxy to on-prem AD), **AWS Managed Microsoft AD** (full MSAD in AWS)
- **AWS Firewall Manager**: centrally manage WAF rules, Shield Advanced protections, Security Groups, and Network Firewall policies across accounts in AWS Organizations
- **AWS Audit Manager**: continuously audits AWS usage to simplify compliance with regulations (HIPAA, GDPR, PCI DSS); produces evidence reports

### 🤖 Amazon Cognito
- **User Pools**: user directory that provides sign-up/sign-in functionality; issues JWT tokens; supports social IdPs (Google, Facebook) and SAML
- **Identity Pools (Federated Identities)**: exchange identity tokens (from User Pools, social IdPs, SAML) for temporary AWS credentials via STS; grants direct access to AWS services

### 📊 Analytics & Data Services
- **Amazon Redshift**: fully managed cloud data warehouse; columnar storage; integrates with S3, Glue, Athena; Redshift Serverless available; Redshift Spectrum queries data directly in S3 without loading
- **Amazon Athena**: serverless interactive query service; queries data in S3 using standard SQL; pay per query (per TB scanned); works with CSV, JSON, ORC, Parquet, Avro; use columnar formats + partitioning to reduce cost
- **AWS Glue**: serverless ETL (Extract, Transform, Load) service; Glue Data Catalog serves as central metadata repository for Athena, Redshift Spectrum, EMR; Glue Crawlers automatically discover schema
- **Amazon OpenSearch Service**: managed OpenSearch/Elasticsearch; full-text search and log analytics; used with Kinesis Firehose for log ingestion pipelines
- **AWS Lake Formation**: simplifies building data lakes on S3; centralizes access control; built on top of Glue
- **Amazon Kinesis** (see above)
- **Amazon MSK (Managed Streaming for Apache Kafka)**: fully managed Apache Kafka; for existing Kafka workloads; unlike Kinesis, you manage broker configuration

### 🧬 ML Managed Services
- **Amazon SageMaker**: build, train, and deploy ML models at scale; managed Jupyter notebooks, built-in algorithms, model hosting
- **Amazon Rekognition**: image/video analysis (object detection, facial recognition, content moderation)

- **Amazon Polly**: text-to-speech

### 🔌 API Gateway
- Fully managed service to create, publish, and secure REST, HTTP, and WebSocket APIs
- Integrates with Lambda (serverless backend), HTTP endpoints, other AWS services
- Stages: deployment environments (dev, staging, prod); supports canary deployments
- Throttling: account-level (10,000 RPS default) and per-stage/method limits; 429 Too Many Requests on breach
- Caching: cache responses by stage; reduces backend calls; configurable TTL
- Authorization: IAM authorization, Lambda authorizers (custom JWT/OAuth validation), Cognito User Pool authorizers

### 💰 EC2 Pricing Models (deep dive)
- On-Demand: highest unit cost, no commitment, good for unpredictable/short-term workloads
- Spot: cheapest (up to 90%), can be interrupted, best for fault-tolerant/stateless workloads (batch, CI/CD, ML training)
- Reserved: 1 or 3-year term; Standard RI = best discount (up to 72%), no flexibility; Convertible RI = slightly lower discount, can change instance family/OS/tenancy; can be sold on Reserved Instance Marketplace (Standard only)
- Savings Plans: commit to $/hour spend; Compute Savings Plans apply to EC2 + Lambda + Fargate; EC2 Instance Savings Plans apply to specific instance family in a region
- Dedicated Host: physical server; useful for BYOL, compliance requirements
- Capacity Reservations: reserve capacity in a specific AZ with no billing discount; combine with RIs/Savings Plans for discount
