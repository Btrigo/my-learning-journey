## 🧠 What I Learned / Key Takeaways
- Port security MAC learning: dynamic (not saved to running config, lost on reboot), static (manually typed, saved), sticky (dynamically learned but written to running config — best of both worlds)
- Violation modes: shutdown (error-disables port, manual recovery), restrict (port stays up, drops traffic, logs it), protect (port stays up, drops traffic, no logging)
- Auto recovery command for shutdown mode: `errdisable recovery cause psecure-violation` — port automatically comes back after a configurable interval but will flap if unauthorized device is still connected
- DHCP starvation attack — Python/Scapy script floods network with DHCP Discover messages using random spoofed MACs, exhausting the entire IP pool so legitimate devices get APIPA addresses instead
- Rogue DHCP server / man in the middle — once pool is exhausted attacker stands up their own DHCP server, hands out IPs with attacker machine as default gateway, intercepts all traffic including clear text credentials via Telnet and HTTP
- DHCP snooping solution — all ports untrusted by default, only port connected to legitimate DHCP server marked trusted, blocks DHCP Offer messages from any untrusted port
- DORA mnemonic — Discover, Offer, Request, Acknowledge (the four DHCP messages)
- Root bridge is elected by lowest priority then lowest MAC as tiebreaker — has nothing to do with proximity to internet or modem
- Spanning tree port states — Blocking → Listening → Learning → Forwarding (~30 seconds total); PortFast skips this on access ports
- Root guard — protects against unauthorized switches becoming root bridge by sending superior BPDUs; puts port into root inconsistent state (not error-disabled); auto-recovers when superior BPDUs stop
- BPDU guard — used on PortFast ports; error-disables port immediately if any BPDU is received; requires manual shut/no shut to recover
- Three tier hierarchical design: core (fast backbone), distribution (routing/policy decisions), access (end device connections)
- Trunk ports carry multiple VLANs using 802.1Q tagging; access ports carry one VLAN only
- Managed switches support VLANs, port security, spanning tree, trunking; unmanaged switches are plug and play with no configuration capability
- Native VLAN travels untagged across trunk links — security best practice is to change it away from VLAN 1, assign it to an unused VLAN, and tag it explicitly with `vlan dot1q tag native`
- Boto3 is AWS's Python SDK — used to interact with AWS services programmatically (Lambda, S3, DynamoDB etc.), not IaC and not a frontend tool
- For Cisco documentation: search "Cisco IOS XE command reference" or "Cisco [feature] configuration guide" on Google — always goes to cisco.com

## 🛠 Commands / Tools Practiced
- `switchport port-security violation restrict` / `protect` / `shutdown`
- `switchport port-security mac-address sticky`
- `errdisable recovery cause psecure-violation`
- `errdisable recovery interval <seconds>`
- `show port-security`, `show port-security interface`, `show port-security address`
- `spanning-tree guard root` — enable root guard on interface
- `spanning-tree portfast` — skip listening/learning on access ports
- `spanning-tree bpduguard enable` — error-disable port if BPDU received
- `show spanning-tree`, `show spanning-tree inconsistentports`
- `show spanning-tree interface gigabitethernet 1/0/1`
- `switchport mode trunk`, `switchport mode access`
- `switchport trunk native vlan <vlan-id>`
- `vlan dot1q tag native` — force native VLAN tagging
- `do show running-config` — run show commands from inside config mode
- `configure terminal` / `conf t` — enter global configuration mode
