## 🧠 What I Learned / Key Takeaways

**Active Directory Basics (room topics — flagged for deeper review, see below)**

- Managing Users in AD / Managing Computers in AD: creating and administering user and computer objects, typically via ADUC
- Group Policies: centralized configuration/enforcement pushed to domain-joined users/computers

- Trees, Forests, and Trusts: how multiple domains relate to and trust one another within a larger AD structure

**Promoting a server to a Domain Controller**
- Process: install the AD DS role (Server Manager or `Install-WindowsFeature AD-Domain-Services -IncludeManagementTools`), then run promotion via Server Manager notification flag or PowerShell (`Install-ADDSDomainController` / `Install-ADDSForest` / `Install-ADDSDomain` depending on scenario)
- Three deployment scenarios: add a DC to an existing domain, add a new domain to an existing forest, or stand up a brand new forest
- DNS Server role is almost always installed alongside AD DS — AD depends on DNS (SRV records) to locate services
- DSRM (Directory Services Restore Mode) password gets set during promotion — ties directly to the "Active Directory repair" Safe Boot option covered in MSConfig's Boot tab a couple sessions back
- `dcpromo.exe` is legacy/deprecated (removed as standalone tool since Server 2012) — now handled through Server Manager/PowerShell
- Common failure points to check: DNS misconfiguration, time sync (Kerberos is time-sensitive), network/firewall connectivity to existing DCs

**Joining a computer to a domain**
- GUI method: System Properties (`sysdm.cpl`) → Computer Name tab → Change → select Domain → enter domain name + credentials → reboot
- PowerShell equivalent: `Add-Computer -DomainName "corp.local" -Credential (Get-Credential) -Restart`
- Prerequisites: client's DNS must point to the domain's DNS server (not public DNS), network connectivity to a DC, valid domain credentials
- Common failure points: DNS resolution failure (most common), time sync drift beyond Kerberos tolerance, duplicate computer object name in AD already existing
- **Key clarification worked through:** ADUC cannot push/force a remote machine to join a domain — the join must be initiated from the client itself, since the client is what authenticates and establishes trust
  - What ADUC CAN do: pre-stage a computer account in a specific OU before the machine joins, reset a broken computer trust relationship (still requires action on the client side after), delete stale computer objects
  - Remote/automated alternatives for real environments: PowerShell remoting (`Invoke-Command` running `Add-Computer` on a remote target), offline domain join (`djoin.exe` — stages join metadata for machines not yet network-connected), or enterprise deployment tools (SCCM, MDT, Intune Autopilot)
