## 🧠 What I Learned / Key Takeaways
- **File separation best practice**: splitting by concern (`providers.tf`, `variables.tf`, `main.tf`, `cloudfront.tf`, `outputs.tf`) is how real teams structure Terraform projects — files are for engineers, not for Terraform
- **HCL block syntax**: every block follows `block_type "label" { }` — curly braces define scope, nesting reflects hierarchy
- **Resource references**: `resource_type.local_name.attribute` (e.g. `aws_s3_bucket.portfolio.arn`) — Terraform resolves dependencies automatically
- **`jsonencode()`**: S3 bucket policies must be submitted as JSON — jsonencode() lets you write in HCL and converts automatically
- **IAM policy structure**: Version, Statement, Effect, Principal, Action, Resource, Condition — keys are capitalized, AWS-defined values must match exactly
- **`providers.tf` first**: provider is the foundation — defines which plugin Terraform uses and how to configure it before any resources are written
- **Type keywords vs string values**: `type = string` has no quotes; `default = "us-east-1"` does — keywords never get quotes, values do
- **Bucket policy Condition block**: `StringEquals` + `AWS:SourceArn` locks access to one specific CloudFront distribution — not any CloudFront, only yours
- **CF Function URI rewriting**: rewrites directory-style requests to index.html before hitting S3 — asset requests (anything with a `.`) are passed through untouched
- **AWS managed cache policy IDs**: hardcoded because AWS owns them, not you — can't reference via Terraform resource syntax
- **HCL data types**: string, number, bool, list, map — all used in this project
