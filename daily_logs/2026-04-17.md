## Key Takeaways

### AWS Global Accelerator
- Routes traffic over **AWS's private global network** instead of the public internet
- Traffic enters at the nearest **edge location**, then stays on AWS's backbone to the endpoint
- Provides **2 static anycast IP addresses** — fixed entry points regardless of backend changes
- Supports endpoints: ALBs, NLBs, EC2 instances, Elastic IPs
- Built-in health checks with near-instant (~30 sec) failover between regions
- **Paid service** — hourly fee per accelerator + per-GB data transfer fee
- Does **NOT** cache content — that's CloudFront's job

**Global Accelerator vs. CloudFront exam trap:**

| | Global Accelerator | CloudFront |
|---|---|---|
| Purpose | Network performance / routing | Content delivery / caching |
| Caches content? | No | Yes |
| Static IPs? | Yes | No |
| Best for | TCP/UDP, gaming, IoT, VoIP, APIs | Static assets, video, websites |
| Layer | Layer 4 (TCP/UDP) | Layer 7 (HTTP/HTTPS) |

> **Trigger:** "static IP addresses" or "non-HTTP traffic" → Global Accelerator. "Caching" or "CDN" → CloudFront.

### VPC Endpoints

**Gateway Endpoints**
- **Free**
- Work with **S3 and DynamoDB only**
- Traffic stays on the AWS private network (no internet required)
- Configured at the **route table** level — no ENI, no private IP

**Interface Endpoints (AWS PrivateLink)**
- **Paid** — hourly + data transfer fees
- Work with **most other AWS services** (SNS, SQS, CloudWatch, EC2 API, etc.)
- Creates an **ENI (Elastic Network Interface)** with a private IP inside your VPC

> **Trigger:** "privately access S3 or DynamoDB from a VPC" → Gateway Endpoint (free). Anything else → Interface Endpoint.

### Reverse Proxy, Load Balancer, and API Gateway

These three are often confused because real-world tools blur the lines. They exist on a **spectrum of capabilities**, not as three completely separate boxes.

**Reverse Proxy — The Foundation**
- Sits **in front of servers** (opposite of a forward proxy, which sits in front of clients)
- Receives requests, forwards to backend, returns response — the client never sees what's behind it
- Use cases: SSL termination, caching, security (hides real server IPs), compression
- Common tools: **Nginx, HAProxy, Caddy, Apache**
- General purpose — doesn't care if it's serving a website, API, or WebSocket

**Load Balancer — Specialized Reverse Proxy**
- One specific mission: **distribute incoming requests across multiple backends**
- Goals: scalability (handle more traffic) and availability (route around failed servers)
- Routing algorithms: Round Robin, Least Connections, IP Hash, Weighted
- **Layer 4 (NLB):** operates at TCP level — sees IPs and ports, fast but limited
- **Layer 7 (ALB):** understands HTTP — can route by URL path, headers, cookies
- AWS offerings: **ALB** (Layer 7), **NLB** (Layer 4)

**API Gateway — API-Aware Reverse Proxy**
- Sits between clients and backends and manages **cross-cutting API concerns**
- Key capabilities:
  - **Authentication & Authorization** — validates tokens once at the edge; invalid requests never reach services
  - **Rate Limiting** — enforces per-client request limits; excess requests rejected at the edge
  - **Request/Response Transformation** — converts formats (JSON ↔ XML), strips sensitive fields
  - **API Versioning** — routes `/v1/users` to old service, `/v2/users` to new service
  - **Analytics & Monitoring** — tracks latency, error rates, most-used endpoints
- Critical in **microservices architectures** — centralizes logic that would otherwise be duplicated across 20+ services
- AWS offering: **AWS API Gateway**

**The Spectrum:**

```
[Reverse Proxy] -----> [Load Balancer] -----> [API Gateway]
Forward requests        + Traffic distribution  + Auth, rate limiting,
Terminate SSL           + Health checks           transformation,
Cache responses         + Failover                analytics
```

**Real-world architecture (typical layered setup):**
1. **CDN (CloudFront/Cloudflare)** — caches static content, terminates SSL at edge
2. **API Gateway** — validates auth, rate limits, routes by URL
3. **Load Balancer** — distributes traffic across service instances, monitors health
4. **Nginx** (optional at service level) — handles SSL for internal traffic, serves static assets

**Decision framework:**
- Multiple instances of a service → **Load Balancer**
- Exposing APIs to external developers → **API Gateway**
- Need auth or rate limiting before hitting services → **API Gateway**
- Just need SSL termination and caching → **Reverse Proxy (Nginx)**
- Often it's a combination of all three

### WebSocket
- **Persistent, full-duplex (two-way) communication** over a single TCP connection
- Unlike HTTP — connection stays open; **either side can send data at any time**
- The server can **push data to the client without the client asking**
- Use cases: live chat, stock tickers, multiplayer games, real-time collaboration
- **AWS API Gateway supports WebSocket APIs**
- Exam trigger: "real-time bidirectional communication" → WebSocket / API Gateway WebSocket API

---

### Nginx
- A **web server** that also functions as a reverse proxy, load balancer, and HTTP cache
- Built to solve the **C10K problem** (10,000+ concurrent connections) using event-driven, non-blocking architecture
- Common roles: serve static files, reverse proxy, SSL termination, load balance, cache responses
- On AWS: **ALB does what Nginx does as a load balancer**, but managed by AWS
