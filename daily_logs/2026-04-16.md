### ☁️ Cloud Benefits (Agility vs Deploy Globally)
- **Agility** = faster development, spin up environments quickly, reduce time to market
- **Deploy Globally in Minutes** = deploying to multiple geographic regions worldwide
- **Trigger:** "time to market / spin up environments faster" → Agility. "Deploy across regions" → Deploy Globally in Minutes
- Elasticity = scale up/down with demand. Cost savings = CapEx → Variable expense

### 💰 CapEx vs OpEx vs Variable
- Cloud shifts **CapEx (upfront fixed cost) → Variable expense (pay-as-you-go)**
- "CapEx traded for OpEx" is partially true but imprecise — the exam wants **Variable expense**
- Variable = you pay only for what you consume, scales with usage

### 🏗️ Core Cloud Infrastructure Categories (NOT "pillars")
- **Compute, Storage, Networking, Databases** — the four foundational cloud resource types
- Do NOT confuse with WAF 6 pillars (OSRPCS) or CAF 6 perspectives

| Storage Class | Designed For | Boot? | Min Storage | Retrieval |
|---|---|---|---|---|
| Standard | Frequently accessed | N/A | None | None |
| Intelligent-Tiering | Unknown/changing patterns | N/A | None | None |
| Standard-IA | Infrequently accessed, rapid retrieval | N/A | 30 days | Per-GB |

### 🖥️ EBS Volume Types (Module 5 — review tomorrow)
| Volume | Type | Boot? | Best For |
|---|---|---|---|
| General Purpose SSD (gp3) | SSD | ✅ Yes | Most workloads, dev/test, balanced |
| Provisioned IOPS SSD (io2) | SSD | ✅ Yes | Critical DBs, sustained high IOPS |
| Throughput Optimized HDD (st1) | HDD | ❌ No | Big data, streaming, log processing |
| Cold HDD (sc1) | HDD | ❌ No | Infrequently accessed, lowest cost |

- **HDDs cannot be used as boot volumes** — only SSDs can
- **IOPS** = Input/Output Operations Per Second — measures how many read/write operations per second
- **Throughput (MiB/s)** = how much data moves per second
- HDDs are not faster than SSDs — they're cheaper and acceptable for large sequential workloads
- SSDs dominate on IOPS; HDDs are chosen for cost savings on sequential/streaming workloads

### 💾 Instance Store
- **Physically attached** to the host server — fastest possible disk performance
- **Temporary/ephemeral** — data is lost when instance stops, hibernates, or terminates
- Device names: `ephemeral0`, `ephemeral1`, etc.
- **Trigger:** "temporary block-level storage / fastest disk / physically attached" → Instance Store
- All other storage options (EBS, EFS, S3) are persistent

### 🗃️ RDS Key Concepts
- **Read Replicas** = actively serve read traffic, offload pressure from master. Up to 15 replicas. Async replication.
- **Multi-AZ Standby** = idle replica in another AZ, only activates on failover. Sync replication. For HA, NOT read throughput.
- **Trigger:** "increase read throughput / read-heavy" → Read Replicas. "High availability / failover" → Multi-AZ
- **BYOL (Bring Your Own License)** = Oracle and MS SQL Server only — commercial/proprietary engines
- **Aurora** = self-healing (data blocks scanned and repaired automatically), high throughput, auto-scales to 64TB, MySQL and PostgreSQL compatible ONLY
- "Self-healing database" → Aurora every time
- RDS does NOT have self-healing — that's Aurora's differentiator

### 🌐 Availability Zones — Why 60 Miles Apart
- Primary reason = **disaster isolation** — a single disaster cannot affect multiple AZs simultaneously
- Secondary constraint = close enough for low-latency fiber connections between AZs
- On the exam: "primary reason" → disaster isolation, not latency optimization

### 🔐 IAM Roles vs IAM Users
- **IAM Role** = temporary permissions, assumed by services, applications, or external entities
  - EC2 needs access to S3 → attach a Role
  - External account/partner needs access → Role (cross-account)
- **IAM User** = permanent human identity, long-term credentials
  - Employee who logs in daily → IAM User
  - Administrator who regularly manages account → IAM User (NOT a role)

### 💻 Programmatic Access to AWS
- Requires **AWS SDK** (the tool) + **Access Keys** (the authentication)
- AWS Management Console = human/browser access, NOT programmatic
- Username/password = Console login, NOT programmatic

### 🛡️ Shared Responsibility Model
- **SQL injection attack on EC2** = customer's responsibility — application-layer security is always the customer's lane
- AWS secures the infrastructure. Customer secures what runs on it.

### 💡 WAF Pillar Trap — Cost Optimization vs Performance Efficiency
- "Right number of resources / right-sizing / eliminate waste" → **Cost Optimization**
- "Best resource type / advanced technology / go serverless" → **Performance Efficiency**
- Both involve resource selection — the differentiator is cost efficiency vs technical performance

### 🌍 Global vs Regional vs Zonal Services
**Global** (no region required):
- IAM, Route 53, CloudFront, AWS WAF, AWS Shield, Organizations, Billing Console

**Zonal** (tied to specific AZ):
- EC2 instances, EBS volumes, Subnets

**Regional** (everything else):
- VPC, S3, Lambda, RDS, DynamoDB, SQS, SNS, CloudTrail, CloudWatch, etc.

### 📡 Route 53 vs CloudFront
- **Route 53** = DNS — translates domain names to IPs, routes traffic. Does NOT serve content.
- **CloudFront** = CDN — actually delivers/serves content from edge locations globally
- **Trigger:** "serve content globally / low latency delivery" → CloudFront. "DNS / domain routing" → Route 53

### 🏢 High Availability Trigger
- "Highly available" → look for **Multi-AZ**
- Spot Instances = opposite of HA (can be interrupted)
- SQS = loose coupling/resiliency, not HA
- CloudFront = performance/CDN, not HA

### 💸 EC2 Billing
- EC2 billed **per second** (60 second minimum) — NOT hourly
- EBS billed per GB **provisioned** per month (not what you use)
- EIP (Elastic IP) = free when attached to running instance, **charged when idle**
- Data transfer: inbound = free, outbound = paid, same-AZ = free, cross-AZ = charged

### 🗂️ AWS Cost & Billing Tools — Critical Distinctions
| Tool | What it does |
|---|---|
| **Cost & Usage Report** | Most granular raw data, loads into Athena/Redshift/QuickSight |
| **Cost Explorer** | Visual interface, custom reports, trends, RI recommendations (lives inside Billing Console) |
| **AWS Budgets** | Set thresholds, alerts when exceeded |
| **Cost Management Dashboard** | High-level month-to-date overview, entry point to all cost tools |

- **Cost Explorer is located inside the AWS Billing Console** — not a standalone service
- "Granular data / load into Athena or Redshift or QuickSight" → Cost & Usage Report specifically

### 🔧 AWS Storage Gateway
- Extends on-prem storage into AWS — local systems see it as local storage
- No data migration required
- **Trigger:** "extend on-prem storage / hybrid storage / no migration needed" → Storage Gateway

### 🛠️ AWS Systems Manager
- Patch Manager: patches EC2 instances without RDP/SSH
- Session Manager: secure shell without opening ports
- Parameter Store: config/secrets storage
- Run Command: execute scripts remotely on instances
- **Trigger:** "patch/manage EC2 without RDP or SSH" → Systems Manager

### 💻 AWS CodeSuite
| Service | Role |
|---|---|
| CodeCommit | Source control / Git repo on AWS |
| CodeBuild | Compile, build, run tests |
| CodeDeploy | Deploy application to EC2/Lambda/on-prem |
| CodePipeline | Orchestrates full CI/CD pipeline end to end |

### 🚀 Elastic Beanstalk vs Lightsail
- **Elastic Beanstalk** = PaaS, developer uploads code, AWS handles infrastructure. Auto-scales. Full AWS ecosystem integration. Production-grade.
- **Lightsail** = simplified VPS for beginners/simple workloads. Fixed monthly pricing. WordPress, small sites.

### 📶 AWS Wavelength
- Embeds AWS compute and storage inside **5G telecom provider datacenters**
- Delivers **single-digit millisecond latency** to mobile devices without leaving the mobile network
- **Trigger:** "5G / mobile applications / ultra-low latency for mobile users" → AWS Wavelength

### 🌐 Consolidated Billing — Volume Discounts
- Pools all accounts' usage together to hit volume pricing tiers faster
- **S3** is the primary example — volume pricing tiers apply to storage
- Also enables Reserved Instance sharing across accounts
- One bill + pooled discounts + RI sharing = the three benefits

### 🖥️ On-Demand Provisioning vs Pay-as-You-Go
- "Servers collecting dust / idle hardware" → **provision resources only when you need them**
- "Pay for only what you use" = billing model concept, not a provisioning concept
- These are related but tested differently on the exam

### 🗄️ DynamoDB Key Facts
- NoSQL key-value store — NOT designed for complex joins or relational queries
- Auto-scales, serverless, millisecond performance at any scale
- 11 9's durability is **S3**, not DynamoDB — common exam distractor
- **Complex joins** = combining multiple related tables in a single query → RDS territory, not DynamoDB
