## 🧠 What I Learned / Key Takeaways
- SFC repairs system files by copying from the local component store (WinSxS); if that store itself is corrupted, SFC has nothing good to copy from — DISM /RestoreHealth fixes the store (pulling from Windows Update), then SFC can succeed. Order matters: DISM first, then SFC.
- 169.254.x.x (APIPA) means the machine never completed a DHCP handshake at all — categorically different from having a real IP but no internet
- Linux service troubleshooting discipline: `systemctl status` (what state), `journalctl -u [service]` (why), fix the actual cause, then restart — don't restart blind
- `ss -tulpn` is the more complete daily-use version (TCP + UDP, listening, process owner, numeric) vs. the narrower `-tlnp` (TCP only)
- Direct parallel: `ss -tulpn` on Linux = `netstat -ano` on Windows — same diagnostic question, different syntax
