## 🧠 What I Learned / Key Takeaways
- **Global vs Account Regional namespace (S3):** Global namespace requires a bucket name unique across every AWS account and region on the planet. Account Regional namespace appends your account ID and region as a suffix, scoping uniqueness to your account only — solves naming conflicts automatically at the cost of a longer bucket name.
- **Why Block Public Access stays ON:** The bucket is private by design. CloudFront accesses it via OAC with signed SigV4 requests — the bucket never talks directly to the public internet. Making the bucket public would bypass CloudFront entirely and expose files to direct S3 URL access.
- **Why Static Website Hosting stays OFF:** OAC requires CloudFront to use the S3 REST endpoint. Static website hosting uses a different endpoint that doesn't support OAC. The two are incompatible — leave it off when CloudFront is in front.
- **Origin path field:** If files are stored in a subfolder (e.g. `/project`), that path goes in the origin path field. Files at bucket root = leave it blank. Putting `/` causes a validation error.
- **CloudFront Function — rewrite vs redirect:** The function rewrites the URI internally before CloudFront processes the request. The visitor never sees it. A redirect would send the user to a different URL — not what's happening here. Naming matters: "rewrite" not "redirect."
- **Why asset requests (CSS, JS, images) are NOT rewritten:** Browsers fire these requests automatically after parsing `index.html`. If the function rewrote `/css/style.css` to `/css/style.css/index.html`, S3 would return a 404 and the page would load broken. Only clean URL paths (no file extension) get rewritten to `/index.html`.
- **Why ACM cert was needed for the live site but not this build:** CloudFront's built-in wildcard cert covers `*.cloudfront.net` only. A custom domain requires a cert that proves domain ownership. ACM DNS validation (adding CNAME records to Route 53) is how you prove ownership — once proven, ACM issues the cert and CloudFront can use it.
- **Bucket policy breakdown:** `Effect: Allow` + `Principal: cloudfront.amazonaws.com` + `Action: s3:GetObject` + `Condition: AWS:SourceArn` matching the specific distribution ARN. Only that one distribution can read from the bucket — any other CloudFront distribution pointed at the bucket gets denied.
- **IAM policy version:** `2012-10-17` is the current policy language version and the correct one to use. `2008-10-17` is legacy. No functional difference for simple policies but `2012-10-17` is best practice.
- **WAF gotcha in new CloudFront wizard:** The new step-by-step CloudFront creation wizard enables WAF by default with a free trial that converts to pay-as-you-go after 14 days. This creates a pricing plan subscription that blocks distribution deletion until the billing cycle ends. Always cancel the WAF plan immediately if not needed, or skip it in the wizard. Plans are per-distribution — cancelling one does not affect other distributions.
- **Teardown order matters:** Disable CloudFront distribution first → wait for it to fully propagate to Disabled → delete distribution → delete S3 objects → delete S3 bucket → delete CloudFront Function. You cannot delete an enabled distribution or a bucket that still has objects in it.
