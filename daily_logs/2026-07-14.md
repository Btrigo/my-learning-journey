## 🧠 What I Learned / Key Takeaways

**Networking / Remote Access**
- Remote connection requires *routability*, not being on the same physical network. Four ways a route gets established: same LAN, VPN (virtual tunnel into the remote network), public IP + port forwarding (also a known ransomware attack vector), or cloud/hosted setups where the provider handles routing behind their own VPN (e.g., TryHackMe → AWS VPC)
- No routing table entry = connection fails immediately, regardless of physical distance — this is exactly what caused the 0x204 RDP error (no OpenVPN tunnel = no route to the lab's private IP)
- Revisited a hazy data-center memory: giving clients a "port number" over the phone during PuTTY sessions was most likely checking the actual configured SSH/RDP port on the target machine (via crash cart + local terminal), not a shared console/terminal server setup
- PuTTY = free Windows SSH client (also does Telnet, rlogin, serial/raw) — historically used because Windows lacked a native SSH client until OpenSSH got bundled into Win10/11

**Windows Fundamentals (system administration)**
- `lusrmgr.msc` (Local Users and Groups) is the single-machine version of what Active Directory does at domain scale — local SAM database per machine, no sync, vs AD's centralized identity/auth (Kerberos) and Group Policy
- On domain-joined machines, `lusrmgr.msc` still shows local accounts (like built-in local Administrator) separate from domain accounts — useful when domain auth itself is broken
- Control Panel maps almost 1:1 to common support tickets (can't print → Devices and Printers; can't connect remotely → Firewall + Remote settings; IP conflict → Network Connections; clock drift → Clock and Region, which breaks Kerberos auth in AD environments)
- Task Manager tabs and their troubleshooting use: Processes/Performance for resource hogs, Startup for boot-time app management (client OS only), Services for a quick status view before dropping into `services.msc`
- **MSConfig (System Configuration)**: General tab (Normal/Diagnostic/Selective startup — Selective is the real troubleshooting tool for isolating a problem via binary search), Services tab ("Hide all Microsoft services" checkbox isolates third-party conflicts), Startup tab is deprecated in favor of Task Manager on client OS — but on Windows Server, neither MSConfig nor Task Manager shows startup items; only `shell:startup` via Run reveals them
- **Boot tab deep dive**: Safe Boot with 4 modes — Minimal (core drivers, no network), Alternate shell (CLI only, no GUI), Active Directory repair (DSRM, DC-only), Network (Minimal + networking). Also covered: No GUI boot, Boot log (`ntbtlog.txt`), Base video (forces generic low-res driver), OS boot information, and the "make permanent" one-way switch. Advanced options button covers CPU/memory capping and kernel Debug settings for deeper crash analysis
- **Advanced System Settings (System Properties, `sysdm.cpl`)**: Computer Name tab (domain/workgroup join), Advanced tab → Performance (pagefile/virtual memory config), User Profiles (delete corrupted profile = classic fix for broken user desktop/settings), Startup and Recovery (auto-restart on crash, memory dump type: Automatic/Kernel/Small/Complete/None), System Protection (System Restore points), Remote tab (Remote Assistance vs Remote Desktop toggle + NLA requirement)
- **Getting into Safe Mode when fully locked out (can't reach login/BSOD loop)**: force 3 hard shutdowns during boot → triggers automatic WinRE → Troubleshoot → Advanced options → Startup Settings → Restart → pick Safe Mode variant via F4/F5/F6. Alternative: boot from Windows installer/recovery USB → "Repair your computer" → same path. Noting the BSOD stop code before rebooting often points directly at the faulting driver
- **Computer Management (`compmgmt.msc`)**: three main sections — System Tools (Task Scheduler, Event Viewer, Shared Folders, Local Users and Groups, Performance, Device Manager), Storage (Windows Server Backup, Disk Management), Services and Applications
