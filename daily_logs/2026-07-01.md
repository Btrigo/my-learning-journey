## 🧠 What I Learned / Key Takeaways

**Group Policy Objects (GPOs)**
- GPOs are settings containers; they do nothing until linked to a scope (OU, domain, site) — linking happens via GPMC by dragging the GPO onto the target OU
- A single GPO can be linked to multiple OUs simultaneously (not one-to-one)
- Linking to the root domain cascades to all child OUs, but Computer Configuration settings are ignored by user-only OUs
- Practiced: created "Restrict Control Panel Access" GPO (User Config), linked to Marketing/Sales/Management; created "Auto Lock Screen" GPO (Computer Config, 5-min inactivity), linked to root domain
- `gpupdate /force` forces immediate policy sync instead of waiting for default refresh window

**Kerberos (deep mechanics)**
- Step 1 — TGT: client sends username + timestamp encrypted with a key derived from the password (proves password knowledge without transmitting it) → KDC returns a TGT (encrypted with the krbtgt account's hash — this is why stealing krbtgt enables Golden Ticket attacks) + a Session Key
- Session Key isn't separately stored by the KDC — it's embedded inside the encrypted TGT and recovered by decrypting it again later
- Step 2 — TGS: client presents TGT + new timestamp encrypted with the Session Key + an SPN (Service Principal Name) identifying the target service → KDC returns a TGS encrypted with the service owner account's hash, containing a Service Session Key
- Step 3: client hands TGS to the actual service, which decrypts it with its own account hash to validate and grant access
- Key insight: every ticket is encrypted with a different party's secret (krbtgt hash for TGT, service account hash for TGS) — this layering is what makes the chain trustworthy without passwords traveling on the network

**NetNTLM (legacy, contrast to Kerberos)**
- No tickets — pure challenge-response: server sends random challenge → client combines it with their NTLM hash to produce a response → for domain accounts, server forwards challenge+response to the DC for verification (DC recalculates and compares)
- Local accounts skip the DC entirely — the machine verifies against its own locally stored hash in the SAM database
- Password/hash itself is never transmitted over the network in either case

**Hashing vs. Encryption**
- Hashing = one-way, irreversible, fixed-length output — used for password storage (never store or transmit plaintext passwords)
- Encryption = two-way, reversible via a key — used when data needs to be read again later by an authorized party (e.g., TGT is encrypted, not hashed, because the KDC needs to decrypt it to recover the Session Key)

**Symmetric vs. Asymmetric Encryption**
- Symmetric: one shared key for encrypt/decrypt, fast, but requires safe key exchange beforehand (e.g., AES) — this is what Kerberos Session Keys and Service Session Keys actually are
- Asymmetric: public/private key pair, no secret exchange needed, but slower (e.g., RSA, ECC) — used for confidentiality (encrypt with recipient's public key) or digital signatures (sign with own private key)
- Real-world pattern: TLS/HTTPS uses asymmetric only for the initial handshake/key exchange, then switches to symmetric (AES) for the bulk data transfer — best of both
- ECC (Elliptic Curve Cryptography) is the modern preferred asymmetric standard over RSA — smaller keys for equivalent security (256-bit ECC ≈ 3072-bit RSA)
- Connected to personal setup: my GitHub SSH keys use Ed25519, which is an ECC-based signing algorithm (EdDSA on Curve25519) — I've already been using ECC without realizing the terminology connection

**Trees, Forests, and Trusts (previously fuzzy — now solid)**
- Tree = domains sharing a contiguous namespace (e.g., thm.local root + uk.thm.local / us.thm.local subdomains), each with its own DC, independently manageable — solves the "need decentralized IT management but stay unified" problem (e.g., company expanding to a new country with different compliance needs)
- Forest = multiple Trees with different namespaces joined together (e.g., after a company merger/acquisition — thm.local + mht.local)
- Enterprise Admins = control over every domain in the entire tree/forest; Domain Admins = control over one domain only — much bigger blast radius if compromised
- Trust relationships: direction of trust is OPPOSITE to direction of access — if Domain AAA trusts Domain BBB, users FROM BBB can access resources ON AAA
- One-way trust = single direction only; two-way trust = mutual (default when domains join a tree/forest)
- Trust ≠ automatic access — it only makes cross-domain authorization possible; specific resource permissions still have to be explicitly granted
- Passed a self-test scenario correctly: identified that if us.thm.local trusts uk.thm.local (one-way), UK users can access US resources
