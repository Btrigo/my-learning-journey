## 🧠 What I Learned / Key Takeaways
- OIDC and OAC are both federated trust relationships, but solve different problems: OAC = CloudFront→S3 trust via SigV4 signing; OIDC = GitHub Actions→AWS trust via signed JWTs
- The `sub` claim in a GitHub OIDC token follows a fixed format: `repo:ORG:REPO:ref:refs/heads/BRANCH` — this is what an IAM trust policy condition checks to scope which repo/branch can assume a role
- `aud` (audience) uses `StringEquals` since it's a single fixed value (`sts.amazonaws.com`); `sub` commonly uses `StringLike` for flexibility, though `StringEquals` is equally valid and arguably stricter for a single-branch use case
- Local folder names and GitHub repo names are completely independent — Git only cares about the remote URL, not the local directory name
- Granting an IAM permission (e.g. `s3:DeleteObject`) is not the same as triggering that behavior — the actual `aws s3 sync` command still needs the `--delete` flag explicitly added for deletions to occur
- Decided to keep CloudFront cache invalidation manual for now (rather than automating it immediately) to build the habit/muscle memory before folding it into the Terraform rebuild phase later — a deliberate "build it twice" tradeoff between best practice and learning reinforcement
