## 🧠 What I Learned / Key Takeaways
- Added a server to a Packet Tracer topology, configured static IP/MAC/gateway/DNS, and used ping to verify full connectivity across all three devices (2 laptops + server)
- Server services in Packet Tracer — HTTP, HTTPS, DHCP, TFTP, DNS, email. Turning a service off breaks connectivity to that port; turning it back on restores it. IIS on Windows is the real-world equivalent of running a web server
- Saving a Packet Tracer topology file preserves your work; link lights go orange on load due to Spanning Tree reconverging before turning green
- Cisco IOS is the operating system of Cisco switches and routers — analogous to Windows, Android, or iOS on other devices. Boots from flash (the switch's equivalent of a hard drive)
- Three CLI modes on a Cisco switch: **User mode** (>) — limited show commands only; **Privileged/Enable mode** (#) — more commands, still can't configure; **Global Configuration mode** (config)# — full configuration access
- Commands to navigate modes: `enable` (user → privileged), `disable` (privileged → user), `conf t` / `configure terminal` (privileged → global config), `end` (global config → privileged), `exit` (steps back one mode)
- Tab autocomplete and abbreviated commands work on Cisco CLI — type enough characters to make the command unique. Ambiguous commands throw an error; `?` shows available options at any point
- `show version` — displays IOS version, uptime, flash info, license info
- `show mac address-table` — displays the switch's learned MAC-to-port mappings (also called CAM table historically)
- `hostname <name>` — sets the switch name, changes the CLI prompt immediately
- `erase startup-config` + `reload` — resets switch to factory defaults
- The switch MAC address table is populated passively — it learns source MAC addresses from incoming frames. No command forces population; you must generate traffic from end devices (ping is the quickest method)
- `arp -a` (PC/server) shows the ARP cache — IP-to-MAC mappings the device has learned. Different from the switch MAC table: PC's perspective (IP→MAC) vs switch's perspective (MAC→port)
- `arp -d` clears the ARP cache — requires admin/elevated privileges on Windows
- **ARP (Address Resolution Protocol)** — when a device knows an IP but not the MAC, it sends a broadcast (destination FF:FF:FF:FF:FF:FF) asking "who has this IP?" The target replies with its MAC. All other devices drop the request. The switch floods the broadcast out all ports. ARP must complete before HTTP or any other traffic can flow
- Packet Tracer simulation mode lets you step through traffic frame by frame and inspect PDU contents at each layer — useful for visualizing ARP, HTTP, and other protocols in action
- **OSI model** created by ISO for standardization and interoperability — solved the proprietary protocol problem (Lightning vs USB-C analogy). 7 layers: Physical, Data Link, Network, Transport, Session, Presentation, Application. Mnemonic: "All People Sleeping Through Networking Don't Pass"
- **TCP/IP won** over OSI in the real world. Focus on the **5-layer hybrid TCP/IP model** for CCNA: Physical, Data Link, Network, Transport, Application
- Real world engineers still say "Layer 7 problem" even though TCP/IP has 5 layers — the numbering comes from OSI
- RFCs (Requests for Comments) — formal documents that define standard protocols. TCP and OSPF are examples
- Layer breakdown:
  - **L1 Physical** — bits, electrical/optical/radio signals. Copper, fiber, WiFi, 5G, Starlink all differ here
  - **L2 Data Link** — frames, MAC addresses, encapsulation per hop. Changes at every router. Ethernet II, PPP, HDLC are different L2 encapsulations
  - **L3 Network** — packets, IP addresses (logical, end-to-end). Router strips and recreates L2 at every hop but IP is end-to-end
  - **L4 Transport** — segments, port numbers, TCP vs UDP. TCP = connection-oriented, three-way handshake, acknowledged (phone call). UDP = connectionless, no confirmation (mailing a letter)
  - **L5-7 Application** — HTTP, HTTPS, FTP, TFTP, SSH, Telnet, SNMP, DNS
- Port numbers live at **Layer 4** — they identify which application on the destination device receives the traffic. The protocol itself (HTTP, FTP, etc.) lives at Layer 7; Layer 4 handles the delivery routing to the right app
- Encapsulation (going down the stack) and decapsulation (going up the stack) — data gets wrapped with headers at each layer on the way out, unwrapped on the way in
- Devices by layer: Hub = L1, Switch/Bridge = L2, Router = L3
- MAC address = physical, per-hop, local network (L2). IP address = logical, end-to-end, routable (L3). Port number = application identifier (L4)
- Key port numbers: HTTP=80, HTTPS=443, SSH=22, Telnet=23, DNS=53, DHCP=67/68, FTP=20/21, TFTP=69, SNMP=161
