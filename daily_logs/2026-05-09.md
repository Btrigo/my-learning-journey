## 🧠 What I Learned / Key Takeaways
- **OAC vs website endpoint tradeoff**: CloudFront's OAC (Origin Access Control) only works with the S3 bucket endpoint, not the website endpoint. The website endpoint doesn't support signed requests. Using the bucket endpoint requires a CloudFront Function to handle index document routing since S3 won't serve index.html automatically on directory requests.
- **CloudFront Function — Viewer request event**: Functions attached to the Viewer request event fire before CloudFront checks its cache or contacts the origin. This is the correct event for URI rewriting — rewrites `/` to `/index.html` and any path without a file extension to append `/index.html`.
- **OAC bucket policy**: The correct bucket policy for OAC uses `Principal: cloudfront.amazonaws.com` with a `Condition` block containing `AWS:SourceArn` pointing to the specific CloudFront distribution ARN. Block public access must be ON when using OAC — S3 is locked down entirely and only CloudFront's signed requests are allowed through.
- **Cache invalidation**: After making configuration changes to a CloudFront distribution, cached responses at edge locations must be invalidated using `/*` to force fresh requests to the origin. Otherwise old cached responses (including error responses) may persist.
- **Kernel updates break out-of-tree drivers**: Ubuntu kernel updates do not automatically rebuild third-party drivers compiled against the previous kernel version. Any driver not in the mainline kernel (NVIDIA, certain WiFi adapters) must be rebuilt against the new kernel. DKMS (Dynamic Kernel Module Support) solves this by hooking into the kernel update process and rebuilding automatically.
- **DFS channels on Linux**: 5GHz WiFi channels 52–140 are DFS (Dynamic Frequency Selection) channels requiring radar detection per regulatory rules. Linux performs passive scanning on these channels, which can cause them to appear completely invisible during quick scans. Windows handles DFS more aggressively. Non-DFS channels 36, 40, 44, 48 are most compatible with Linux.
- **apt purge vs apt remove**: `apt remove` uninstalls a package but keeps config files on disk. `apt purge` removes the package AND all associated config files. Use `purge` when reinstalling broken drivers to guarantee a clean slate — leftover configs from a broken install can cause the reinstall to fail or behave unexpectedly.
- **Static website hosting and OAC conflict**: Having S3 static website hosting enabled on a bucket while also using the bucket endpoint with OAC may cause conflicts. The website hosting feature is designed for the website endpoint. When using OAC + bucket endpoint, static website hosting may need to be disabled since routing is handled by the CloudFront Function instead.

## 🛠 Commands / Tools Practiced
- `nvidia-smi` — verify NVIDIA driver communication with GPU
- `xrandr --query` — query connected displays and their resolutions
- `ubuntu-drivers devices` — list available drivers for detected hardware
- `sudo apt purge nvidia-* -y` — completely remove all NVIDIA packages including configs
- `sudo apt autoremove -y` — remove orphaned dependencies
- `sudo apt install nvidia-driver-535 -y` — install LTS NVIDIA driver
- `iw list | grep -A 10 "Frequencies:"` — check adapter's supported frequency bands
- `nmcli dev wifi list` — list visible WiFi networks
- `nmcli dev wifi rescan` — force active rescan of all frequencies
- `sudo iw dev [interface] scan | grep -E "SSID|freq"` — raw driver-level WiFi scan
- `iwconfig` — display wireless interface statistics including bit rate and signal strength
- `lsmod | grep 88x2bu` — check if Realtek WiFi driver module is loaded in kernel
- `git clone` — clone driver source from GitHub
- `sudo ./install-driver.sh` — install driver via DKMS
- AWS Console: CloudFront OAC creation, behavior editing, function association, cache invalidation
